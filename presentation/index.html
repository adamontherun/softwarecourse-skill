The mechanism

The whole thing is
one markdown file

No framework, no pipeline code. A skill is a written playbook that Claude Code reads when the task matches.

Just under 800 lines of process, most of it earned the hard way: a mobile bug found after shipping, a lint retrofit that surfaced 56 violations, a fake package that almost got installed. Each lesson became an instruction.

The process

One git commit per stage. A course you can review, resume, or roll back.

Ground rule

Nothing load-bearing
comes from memory

  • Default values, gotchas, version claims: researched against current docs, cited inline
  • Output shown in the book is captured from code that actually ran
  • Citations get re-fetched later to confirm the page still says what the book claims
why it matters A fabricated number looks exactly as credible as a real one. The reader only finds out in production.

GPT runs headless through Cursor's CLI. It never sees Claude's verdicts, so it can't rubber-stamp them.

Settling a dispute

Contested claims get
a real adjudication

  • Fresh agents research the claim with no knowledge of either verdict
  • A documented API page beats a blog post, whichever model cited it
  • Every correction lands in its own commit, with the second model named in the message
tested for real We fed it a planted lie: "requests times out after 5 seconds by default." GPT came back DISPUTED in 16 seconds, with the docs page proving requests never times out unless you ask.
The finish line

Done means it ran

  • Challenge tests must fail on the empty skeleton and pass on the solution
  • CI runs both directions on every push, plus a devcontainer smoke test
  • The deployed site gets fetched and checked before anyone declares victory

A green checkmark is a claim too. The skill treats it like any other: verify it by looking at the log.
